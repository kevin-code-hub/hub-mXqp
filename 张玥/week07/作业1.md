# 作业1：阅读 `02-joint-bert-training-only` 代码分析

## 题目

阅读 `02-joint-bert-training-only` 代码，并回答以下问题：

1. BERT 文本分类和实体识别有什么关系，分别使用什么 loss？
2. 多任务训练 `loss = seq_loss + token_loss` 有什么坏处？如果存在训练不平衡的情况，如何处理？

---

## 一、BERT 文本分类和实体识别有什么关系？分别使用什么 loss？

### 1.1 模型结构上的关系

从模型结构上看，这个项目采用的是一种**多任务学习（Multi-Task Learning）**设计，即：

* **共享一个 BERT 编码器**
* 在 BERT 输出之上接两个不同的任务头（task head）

具体来说：

* `pooler_output` 用于做**句级 intent 分类**
* `token_output` 用于做**token 级 slot filling / NER 标注**

也就是说，这两个任务之间的关系不是“前一个任务输出作为后一个任务输入”的串联关系，而是：

> **共享底层语义表示的硬参数共享（hard parameter sharing）关系。**

---

### 1.2 两个任务分别解决什么问题？

可以从任务目标上理解它们的区别：

#### （1）intent 分类

回答的是：

> **这句话整体想表达什么意图？**

这是一个**句子级分类任务**，关注全局语义。

例如一句话：

> “帮我订一张明天去上海的机票”

intent 可能是：

* `book_flight`

---

#### （2）slot filling / NER

回答的是：

> **这句话里哪些词是关键参数？它们分别属于什么槽位或实体类型？**

这是一个**token 级序列标注任务**，更关注局部词语和边界信息。

例如：

* “明天” → `B-date`
* “上海” → `B-city`

---

### 1.3 二者为什么可以一起训练？

这两个任务通常具有一定的互补性：

* **intent 会约束 slot 的可能范围**
  例如，如果 intent 是“订机票”，那么更容易出现“出发地、目的地、时间”等槽位。

* **slot 也会反过来帮助 intent 判断**
  例如，如果一句话中识别出了“城市、日期、航班”等信息，那么 intent 很可能是“机票预订”。

不过，需要注意的是：

> **这份代码里这种互相帮助并不是通过显式建模实现的，而是通过共享 BERT 隐层表示间接实现的。**

也就是说，这个项目**没有显式建模 `domain-intent-slot` 的层级依赖关系**，只是让两个任务共享同一个语义编码器。

---

## 二、分别使用什么 loss？

这个项目里两个任务都使用了 `nn.CrossEntropyLoss()`，但作用对象不同。

### 2.1 intent 分类使用的 loss

intent 分类属于**句级多分类任务**，对应的 loss 是：

```python
seq_loss = CrossEntropyLoss(seq_output, seq_label_ids)
```

其作用是：

* `seq_output`：句子级分类 logits
* `seq_label_ids`：对应的 intent 标签

本质上就是：

> **对整句话做多分类交叉熵损失。**

---

### 2.2 token / NER 使用的 loss

token 级实体识别使用的 loss 是：

```python
token_loss = CrossEntropyLoss(active_logits, active_labels)
```

它的处理逻辑是：

1. 先利用 `attention_mask` 过滤掉 padding 位置
2. 只在有效 token 上计算 loss

因此它本质上是：

> **只对非 padding 的有效 token 做多分类交叉熵损失。**

---

### 2.3 标签形式

该项目中的 slot / NER 标签体系使用的是 **BIO 标注格式**，即：

* `O`：不属于任何实体
* `B-*`：某类实体的开始
* `I-*`：某类实体的内部

例如：

* `B-city`
* `I-city`
* `B-date`

这种 BIO 扩展方式是在 `config.py` 中根据槽位名生成的。

---

## 三、`loss = seq_loss + token_loss` 有什么坏处？

该项目在训练时直接将两个任务的 loss 相加：

```python
loss = seq_loss + token_loss
```

这种写法简单直接，但也存在明显问题。

### 3.1 任务权重被硬编码为 1:1

虽然表面上两个 loss 都是“加 1 倍”，但这并不代表两个任务对共享 BERT 的训练贡献真的平衡。

原因在于：

* 两个 loss 的数值量级可能不同
* 即使 loss 数值接近，反传到共享层后的**梯度范数**也可能差异很大

结果就是：

> **某一个任务可能主导训练，另一个任务学不动。**

---

### 3.2 容易出现负迁移（Negative Transfer）

两个任务虽然相关，但优化目标并不完全一致：

* intent 分类偏向**全局语义**
* slot filling / NER 偏向**局部边界与标签一致性**

它们对共享表示层的最优方向不一定一致。
如果直接等权相加，可能会出现：

> **两个任务在共享层上互相拉扯，彼此干扰。**

---

### 3.3 不同任务的收敛速度不同

一般来说：

* intent 分类通常更容易收敛
* slot filling / NER 往往更难学

如果一直简单相加，可能会出现两种情况：

* intent 已经学得很好，但 NER 还没学好
* 或者反过来，一个任务拖慢另一个任务

这会导致训练过程不稳定，或者最终结果不均衡。

---

### 3.4 数据分布不平衡时问题更明显

如果数据本身存在不平衡，例如：

* 某些 intent 样本很多，某些很少
* 某些槽位标签极少
* 一句话很短，导致 token 监督信号稀疏
* NER 中 `O` 标签数量远多于实体标签

那么直接相加不会自动修正这些偏差，模型就可能更偏向“样本更多、监督更强”的那一侧任务。

---

## 四、如果存在训练不平衡，如何处理？

如果训练中确实出现多任务不平衡问题，常见的处理方法如下。

### 4.1 加权求和（最直接、最实用）

最简单的方法是引入任务权重：

```python
loss = alpha * seq_loss + beta * token_loss
```

例如可以尝试：

* `alpha = 1, beta = 2`
* `alpha = 0.5, beta = 1`

然后在验证集上搜索较优组合。

这种方法的优点是：

* 实现简单
* 成本低
* 通常很快就能改善训练失衡问题

这是最推荐的第一步方案。

---

### 4.2 动态调整任务权重

相比固定权重，更进一步的方法是根据训练过程动态调整权重。

常见方法有：

* **Uncertainty Weighting**
* **GradNorm**
* **DWA（Dynamic Weight Average）**

它们的核心思想是：

> 根据任务难度、收敛速度或梯度大小，自动调节各任务的 loss 权重。

这种方法通常比手工调权重更稳，但实现复杂度也更高。

---

### 4.3 处理梯度冲突

有时候问题不在于 loss 数值，而在于两个任务在共享层上的**梯度方向冲突**。

这时可以考虑专门的多任务梯度优化方法，例如：

* **PCGrad**
* **MGDA**

它们会在参数更新前先处理冲突梯度，减少任务之间的相互伤害。

适合于：

* 多任务差异较大
* 明显出现负迁移
* 单纯调权重效果有限

---

### 4.4 分阶段训练或交替训练

如果两个任务差异较大，也可以采用训练策略上的调整。

例如：

#### 方案 1：分阶段训练

* 先单独训练 intent 头
* 再联合训练 intent + slot

#### 方案 2：交替训练

* 某些 step 更强调 intent
* 某些 step 更强调 slot

这种方式在：

* 数据量较少
* 任务差异较大
* 联合训练不稳定

的情况下比较常见。

---

### 4.5 对单个任务内部做类别不平衡处理

除了任务间不平衡，还可能存在任务内部标签分布不平衡。

可以考虑：

#### 对 intent 分类：

* 在交叉熵中加入 `class weight`

#### 对 slot / NER：

* 按标签频次加权
* 如果 `O` 标签过多，可以考虑 **Focal Loss**

这样做的目的是：

> 避免模型被高频类别主导，忽略低频但重要的标签。

---

### 4.6 增强 NER 头结构

当前项目中的 token 头，本质上还是：

> **BERT 输出 + 逐 token 线性分类**

这种方式实现简单，但对边界约束较弱。

如果发现 NER 的边界学习效果不好，常见升级方案是：

* **BERT + CRF**

CRF 的好处在于：

* 能显式建模标签转移约束
* 减少非法 BIO 序列
* 提升序列标注稳定性

需要注意的是：

> 这不一定能直接解决多任务不平衡，但通常能提升 token 任务本身的效果和稳定性。

---

### 4.7 显式加入 domain 任务

从任务设计角度看，这份代码里其实没有显式使用 `domain` 信息。

如果数据中 `domain` 和 `intent` 有较强相关性，那么可以考虑：

* 再增加一个 `domain classification head`
* 或者采用层级预测：先 domain，再 intent，再 slot

这样能让共享表示更贴近实际任务结构。

不过代价是：

* 多了第三个任务
* 需要再引入新的 loss
* 训练平衡问题也会进一步复杂化

---

## 五、总结

### 5.1 关于“文本分类和实体识别的关系”

这个项目中的 intent 分类和 slot filling / NER：

* 共享同一个 BERT 编码器
* 分别接句级分类头和 token 级标注头
* 属于典型的多任务学习结构

二者并不是串联关系，而是：

> **共享底层语义表示的并行任务关系。**

intent 负责回答“这句话想干什么”，slot/NER 负责回答“这句话中的关键参数是什么”。

---

### 5.2 关于“分别使用什么 loss”

二者都使用 `CrossEntropyLoss`：

* **intent 分类**：句级多分类交叉熵
* **slot / NER**：有效 token 上的多分类交叉熵

其中 NER 标签采用 BIO 体系。

---

### 5.3 关于“直接相加 loss 的问题”

`loss = seq_loss + token_loss` 的主要问题有：

1. 任务权重被硬编码为 1:1
2. 容易出现负迁移
3. 两个任务收敛速度不同
4. 数据分布不平衡时问题会更严重

---

### 5.4 关于“如何处理训练不平衡”

更实用的处理顺序建议是：

1. **先做加权求和**
2. **再考虑动态权重**
3. **必要时处理梯度冲突**
4. **结合分阶段训练、类别加权、CRF 增强等方法进一步优化**

---

## 六、结论

该项目中，intent 分类和 slot filling / NER 采用的是**共享同一个 BERT 编码器、上层分接两个任务头**的多任务学习结构。二者不是串联关系，而是共享底层语义表示的并行关系。intent 解决句子整体意图识别，slot/NER 解决 token 级关键信息抽取，它们可以通过共享表示相互促进，但代码中并没有显式建模 domain-intent-slot 的依赖关系。

loss 方面，两个任务都使用 `CrossEntropyLoss`。其中 intent 分类使用句级分类交叉熵，slot/NER 使用有效 token 上的多分类交叉熵，并结合 `attention_mask` 过滤 padding 位置。NER 标签采用 BIO 体系。

直接使用 `loss = seq_loss + token_loss` 的问题在于：任务权重被硬编码为 1:1，容易导致梯度贡献失衡；两个任务目标不同，可能产生负迁移；同时不同任务的收敛速度和数据分布不一致，也会加剧训练不平衡。实际处理中，最常见的做法是先采用加权求和，再进一步考虑动态权重、梯度冲突处理、分阶段训练、类别加权，以及对 NER 头引入 CRF 等增强方法。
